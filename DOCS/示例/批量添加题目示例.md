## 一、功能概述（已验证）

### 1.1 实际工作原理

批量添加题目的**实际实现方式**是：

```
文本输入 → 前端解析( BatchAddQuestion.vue ) → 逐条调用 POST /api/questions → 循环插入
```

**关键事实（基于代码验证）**：
- ❌ **没有**专用的批量导入 API 端点
- ❌ **不支持** Excel (.xlsx) 文件上传
- ✅ 仅支持**文本粘贴解析**方式
- ✅ 解析后通过 `POST /api/questions` **逐条创建**每道题目
- ✅ 前端组件：[BatchAddQuestion.vue](../../src/components/admin/question-management/BatchAddQuestion.vue)
- ✅ 后端处理：[questions.js](../../routes/questions.js) 的 `POST /` 路由

### 1.2 支持的题型

| 题型标识 | 中文名称 | 选项要求 | 答案格式 |
|----------|----------|----------|----------|
| `single` | 单选题 | 2-6个选项 | 单个字母 A/B/C/D |
| `multiple` | 多选题 | 3-8个选项 | 多个字母如 ABC |
| `judgment` | 判断题 | 固定 `["对", "错"]` | A=对 B=错 |

### 1.3 使用限制

- 数量限制：无硬性后端限制（建议单次 ≤50 道，避免请求超时）
- 每道题目必填字段：`subjectId`, `subcategoryId`, `content`, `type`, `options`, `answer`

## 二、文本格式导入（唯一支持的方式）

### 2.1 支持的文本格式

系统通过正则表达式解析以下两种格式的文本：

#### 格式一：带编号的题目（推荐）

```
1. 下列哪个是 JavaScript 的基本数据类型？(A)
A. String
B. Number
C. Boolean
D. Object
解析：JavaScript 有 7 种基本数据类型。

2. Vue 3 中用于创建响应式数据的 API 是？(B)
A. data()
B. ref()
C. reactive()
D. state()

3. 【判断】JavaScript 是一种强类型语言。(×)
```

#### 格式二：不带编号的题目

```
下列哪个是 JavaScript 的基本数据类型？(A)
A. String
B. Number
C. Boolean
D. Object

【判断】HTTP协议中POST请求参数显示在URL中。(√)
```

### 2.2 格式规则详解

#### 题目行格式

```
[编号.] 题目内容(答案字母)

或

【判断】题目内容。(√或×)
```

**答案括号说明**：
- `(A)` `(B)` `(C)` `(D)` → 普通题，对应选项字母
- `(√)` 或 `(×)` → 判断题，√=对(A) ×=错(B)

#### 选项行格式

```
A. 选项内容
B. 选项内容
C. 选项内容
```

- 以大写字母 + 英文句点/中文顿号开头
- 选项内容紧跟标点后

#### 解析行格式（可选）

```
解析：这里是解析内容。
```

### 2.3 完整文本示例（含三种题型）

```
1. 下列哪个是 JavaScript 的基本数据类型？(A)
A. String
B. Number
C. Boolean
D. Object
解析：String、Number、Boolean、Null、Undefined、Symbol、BigInt 是基本类型。

2. 以下哪些是 JavaScript 的循环语句？(ABC)
【多选】
A. for循环
B. while循环
C. do...while循环
D. if语句
E. switch语句
解析：if和switch是条件语句，不是循环语句。

3. 【判断】JavaScript 是一种强类型语言。(×)
解析：JavaScript 是弱类型（动态类型）语言。

4. 在 HTTP 协议中，GET 请求的参数会显示在 URL 中。(√)
解析：GET 参数通过 Query String 传递。
```

## 三、实际 API 接口（已验证）

### 3.1 单条添加接口

批量添加的底层实际调用的是**单条添加接口**：

| 属性 | 值 |
|------|-----|
| **Endpoint** | `POST /api/questions` |
| **Content-Type** | `application/json` |
| **认证要求** | 无需（题目管理接口当前无需认证） |

#### 请求体结构（基于 [questions.js L349-472](../../routes/questions.js#L349-L472)）

```json
{
  "subjectId": 1,
  "subcategoryId": 1,
  "content": "题目内容",
  "type": "single",
  "options": ["选项A", "选项B", "选项C", "选项D"],
  "answer": "A",
  "explanation": "解析内容",
  "audio": null,
  "image": null
}
```

#### 字段详细说明（基于数据库表结构验证）

| 字段名 | 类型 | 必填 | 数据库字段 | 说明 |
|--------|------|------|------------|------|
| `subjectId` | number | **是** | `subject_id` INT | 学科 ID（必须是有效值） |
| `subcategoryId` | number | **是** | `subcategory_id` INT | 子分类 ID（必须是有效值） |
| `content` | string | **是** | `content` LONGTEXT | 题干内容（支持 HTML 富文本） |
| `type` | string | **是** | `type` VARCHAR(50) | 题型枚举：`single` / `multiple` / `judgment` / `reading` |
| `options` | array | **是** | `options` LONGTEXT | 选项数组（JSON 序列化存储） |
| `answer` | string | **是** | `correct_answer` TEXT | 正确答案（A/B/C/D 或 AB,CD 等） |
| `explanation` | string | 否 | `explanation` TEXT | 解析内容 |
| `audio` | string \| null | 否 | `audio_url` VARCHAR(255) | 音频文件 URL |
| `image` | string \| null | 否 | `image_url` VARCHAR(255) | 图片文件 URL |

> ⚠️ **注意**：API 使用 **camelCase** 字段名（`subjectId`），非 snake_case（`subject_id`）
>
> ⚠️ **注意**：**没有** `difficulty`、`tags`、`analysis` 等字段！这些在创建/编辑接口中不存在

#### 响应示例 (200)

```json
{
  "id": 461,
  "subjectId": 1,
  "subcategoryId": 1,
  "content": "<p>1+1等于多少？</p>",
  "type": "single",
  "options": ["A. 1", "B. 2", "C. 3", "D. 4"],
  "answer": "B",
  "explanation": "这是一道基础加法题",
  "audio": null,
  "image": null,
  "createdAt": "2026-04-05T12:00:00.000Z"
}
```

#### 错误响应

```json
{ "error": "题目信息不完整" }                              // 400 - 必填字段缺失
{ "error": "判断题选项必须为[\"对\", \"错\"]" }              // 400 - 判断题校验失败
{ "error": "判断题答案必须为 A 或 B" }                       // 400 - 判断题答案错误
{ "error": "添加题目失败", "details": "..." }                // 500 - 数据库错误
```

### 3.2 批量操作接口（非导入！）

> ⚠️ 此接口用于**操作已有题目**，不是导入新题目！

| 属性 | 值 |
|------|-----|
| **Endpoint** | `POST /api/questions/batch` |
| **功能** | 批量删除 / 修改难度 / 修改题型 / 移动分类 |

#### 支持的操作类型

```json
// 批量删除
{ "action": "delete", "ids": [1, 2, 3] }

// 批量修改难度
{ "action": "updateDifficulty", "ids": [1, 2], "data": { "difficulty": 3 } }

// 批量修改题型
{ "action": "updateType", "ids": [1, 2], "data": { "type": "multiple" } }

// 批量移动到其他学科
{ "action": "move", "ids": [1, 2, 3], "data": { "subjectId": 2, "subcategoryId": 5 } }
```

**限制**：单次最多操作 100 条记录

## 四、前端使用流程（已验证）

基于 [BatchAddQuestion.vue](../../src/components/admin/question-management/BatchAddQuestion.vue) 和 [AdminView.vue](../../src/views/AdminView.vue#L224-L242) 的实际代码：

### 步骤 1：打开批量添加对话框

在管理后台 → 题目管理页面 → 点击「批量添加」按钮

### 步骤 2：选择目标学科和子分类

- 左侧下拉框选择**学科**
- 第二个下拉框选择**学科题库**（子分类）

### 步骤 3：粘贴题目文本

将按【第二章】格式编写的题目文本粘贴到文本框中

支持的界面提示：
```
请粘贴题目文本，支持以下格式：

单选题/多选题：
1. 题目内容(A)
A. 选项1
B. 选项2
C. 选项3
D. 选项4

判断题：
1. 题目内容(√)
或
【判断】题目内容。(×)
```

### 步骤 4：点击「解析题目」

系统自动解析文本并在右侧**预览区域**展示解析结果：
- 显示题目数量
- 显示每道题的 content、options、answer、explanation
- 正确答案会标记绿色标签

### 步骤 5：点击「批量添加」

触发流程：
1. `BatchAddQuestion.vue` 发出 `batch-add-questions` 事件，携带解析后的题目数组
2. `AdminView.vue` 的 `handleBatchAddQuestions` 方法接收事件
3. **逐条循环调用** `questionStore.addQuestion(question)`
4. 每条调用发送 `POST /api/questions` 创建一道题目
5. 全部完成后显示成功消息：「成功添加 N 道题目」
6. 自动刷新题目列表

## 五、各题型详细示例

### 5.1 单选题示例

**文本输入**：
```
1. 1+1等于多少？(B)
A. 1
B. 2
C. 3
D. 4
解析：基础加法运算。
```

**解析后的数据**（将发送给 API）：
```json
{
  "subjectId": 1,
  "subcategoryId": 1,
  "type": "single",
  "content": "1+1等于多少？(B)",
  "options": ["1", "2", "3", "4"],
  "answer": "B",
  "explanation": "基础加法运算。"
}
```

### 5.2 多选题示例

**文本输入**：
```
1. 【多选】以下哪些是前端框架？(ACD)
A. React
B. Python
C. Vue
D. Angular
E. Java
解析：React、Vue、Angular 是前端框架。
```

**解析后的数据**：
```json
{
  "subjectId": 1,
  "subcategoryId": 1,
  "type": "multiple",
  "content": "【多选】以下哪些是前端框架？(ACD)",
  "options": ["React", "Python", "Vue", "Angular", "Java"],
  "answer": "ACD",
  "explanation": "React、Vue、Angular 是前端框架。"
}
```

> 多选题识别规则：答案包含多个字母（长度 > 1）或题目含【多选】标记

### 5.3 判断题示例

**文本输入**（两种写法均可）：
```
1. 【判断】地球绕着太阳转。(√)

2. 月亮本身会发光。(×)
```

**解析后的数据**：
```json
{
  "subjectId": 1,
  "subcategoryId": 1,
  "type": "judgment",
  "content": "地球绕着太阳转。",
  "options": ["对", "错"],
  "answer": "A",
  "explanation": ""
}
```

```json
{
  "subjectId": 1,
  "subcategoryId": 1,
  "type": "judgment",
  "content": "月亮本身会发光。",
  "options": ["对", "错"],
  "answer": "B",
  "explanation": ""
}
```

> 判断题识别规则：题目含【判断】标记 或 答案括号内为 √/×
>
> √ → answer = "A"（对）| × → answer = "B"（错）
>
> 选项固定为 `["对", "错"]`，后端会强制校验

## 六、注意事项与常见问题

### 6.1 关键注意事项

| 注意事项 | 说明 | 后果 |
|----------|------|------|
| **subjectId 有效性** | 必须是数据库中存在的学科 ID | 插入失败或关联错误数据 |
| **subcategoryId 有效性** | 必须是对应学科下的分类 ID | 同上 |
| **判断题选项固定** | 必须为 `["对", "错"]` | 后端返回 400 错误 |
| **判断题答案范围** | 只能是 `A` 或 `B` | 后端返回 400 错误 |
| **答案使用大写字母** | A/B/C/D | 小写可能导致匹配异常 |
| **XSS 自动过滤** | content 和 explanation 会经过 XSS 清理 | HTML 标签可能被移除 |
| **Base64 图片转换** | 内嵌 Base64 图片会被自动转为文件存储 | 无需手动处理 |

### 6.2 常见错误及解决方案

#### 错误 1: 解析后题目数量不对

**原因**: 文本格式不符合规范

**解决**: 检查以下要点：
- 每行题目必须有答案括号 `(A)` 或 `(√)`
- 选项行必须以 `A.` `B.` 等字母开头
- 题目之间用空行分隔（可选但推荐）
- 不要有多余的空行在选项中间

#### 错误 2: 判断题解析失败

**原因**: 格式不符合判断题识别规则

**解决**: 使用以下任一格式：
```
✅ 正确格式：
1. 【判断】这是陈述内容。(√)
1. 这是陈述内容。(×)

❌ 错误格式：
1. 这是对的还是错的？
A. 对
B. 错
```

#### 错误 3: 添加过程中部分失败

**原因**: 逐条调用时某条题目数据不合法

**解决**:
- 当前实现：任何一条失败会导致整个批次的后续题目停止添加
- 建议：先少量测试（2-3道），确认格式正确后再大批量添加
- 检查浏览器控制台查看具体哪条失败

#### 错误 4: 选项顺序与答案不匹配

**原因**: 文本中选项顺序与标准 A/B/C/D 不一致

**解决**: 确保选项按 A→B→C→D 顺序排列，答案字母指向正确的选项

## 七、最佳实践

### 7.1 数据准备建议

1. **先用少量测试**（2-3道题）确认格式正确
2. **分批次导入**（每批 ≤50 道）避免超时
3. **检查预览结果**：点击「解析题目」后仔细核对右侧预览
4. **备份原始文本**：防止解析出错需要重新输入

### 7.2 文本编写技巧

```
✅ 推荐：
- 每道题目前加编号 1. 2. 3.
- 选项前统一用 A. B. C. D.
- 判断题使用【判断】标记
- 答案括号紧跟题目内容
- 解析以「解析：」开头（可选）

❌ 避免：
- 选项使用 ① ② ③ ④ 等特殊符号
- 答案括号与题目之间有过多空格
- 混合使用不同编号风格
- 在选项行中混入非选项内容
```

## 附录：实际数据库 questions 表结构

```sql
CREATE TABLE `questions` (
  `id` int NOT NULL AUTO_INCREMENT,
  `subject_id` int NOT NULL,
  `subcategory_id` int NOT NULL,
  `content` longtext NOT NULL,
  `type` varchar(50) NOT NULL,
  `options` longtext NOT NULL,
  `correct_answer` text NOT NULL,
  `explanation` text,
  `audio_url` varchar(255) DEFAULT NULL,
  `image_url` varchar(255) DEFAULT NULL,
  `difficulty` int DEFAULT '1',
  `created_at` datetime DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  KEY `idx_subject_id` (`subject_id`),
  KEY `idx_subcategory_id` (`subcategory_id`),
  KEY `idx_type` (`type`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

> 注意：`difficulty` 字段存在于数据库中，但 **创建/编辑 API 不接收此参数**（由难度自适应服务自动管理）
