# 🚨 部署后报错快速修复指南

## 问题现象

部署 AItest 分支后，前端报错：

```
GET http://10.78.127.30:3001/api/chat/models?page=1&limit=12 500 (Internal Server Error)
GET http://10.78.127.30:3001/api/chat/models/stats 500 (Internal Server Error)
POST http://10.78.127.30:3001/api/chat/models 500 (Internal Server Error)
```

---

## 问题原因

**数据库缺少 AI 聊天功能所需的表**：

- `ai_models` - AI 模型配置表 ❌ 不存在
- `chat_sessions` - 聊天会话表 ❌ 不存在
- `chat_messages` - 聊天消息表 ❌ 不存在
- `token_usage` - Token 使用记录表 ❌ 不存在
- `cache_hits` - 缓存命中记录表 ❌ 不存在

---

## ⚡ 快速修复步骤（5分钟）

### 第一步：SSH 登录服务器

```bash
ssh user@your-server-ip
cd /var/www/PSCG
```

---

### 第二步：备份数据库

```bash
# 备份数据库
mysqldump -u root -p --single-transaction pscg_db > ~/backups/pscg_db_backup_$(date +%Y%m%d_%H%M%S).sql

# 验证备份
ls -lh ~/backups/
```

---

### 第三步：执行数据库迁移

**方法1：使用 Git 拉取迁移脚本**

```bash
# 拉取最新代码（包含迁移脚本）
git pull origin AItest

# 执行迁移脚本
mysql -u root -p pscg_db < migrations/migrate_to_aittest.sql
```

**方法2：手动上传迁移脚本**

如果 Git 拉取失败，可以手动上传：

```bash
# 在本地 Windows PowerShell 执行
scp migrations/migrate_to_aittest.sql user@your-server:/tmp/

# 在服务器执行
mysql -u root -p pscg_db < /tmp/migrate_to_aittest.sql
```

---

### 第四步：验证数据库表

```bash
# 检查表是否创建成功
mysql -u root -p pscg_db -e "SHOW TABLES LIKE 'ai_%'"

# 应该输出：
# ai_models
# chat_messages
# chat_sessions

# 检查 token_usage 和 cache_hits 表
mysql -u root -p pscg_db -e "SHOW TABLES LIKE '%usage'"

# 应该输出：
# token_usage

mysql -u root -p pscg_db -e "SHOW TABLES LIKE '%hits'"

# 应该输出：
# cache_hits
```

---

### 第五步：重启服务

```bash
# 重启 PM2 服务
pm2 restart PSCG

# 查看日志
pm2 logs PSCG --lines 50

# 健康检查
curl http://localhost:3001/api/health
```

---

### 第六步：测试 AI 聊天功能

在浏览器中：

1. 登录后台管理
2. 进入"AI 模型管理"页面
3. 应该能看到空列表或示例模型
4. 点击"添加模型"测试功能

## ⚠️ 注意事项

1. **备份数据库**：执行迁移前务必备份数据库
2. **迁移脚本幂等**：可以安全重复执行，不会覆盖现有数据
3. **后续配置**：迁移完成后需要在后台配置 AI 模型的 API Key
4. **回滚方案**：如有问题，可执行 `rollback_from_aittest.sql` 回滚

## 📞 遇到问题？

如果上述步骤无法解决问题：

1. 检查服务日志：`pm2 logs PSCG --lines 100`
2. 检查数据库日志：`tail -f /var/log/mysql/error.log`
3. 手动验证表结构：`mysql -u root -p pscg_db -e "DESC ai_models"`
4. 联系开发团队并提供完整日志
