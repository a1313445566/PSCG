## 一、强制验证流程（5步法）

在进行任何文档编写、重构或更新之前，**必须**严格按照以下 5 步验证流程执行。违反此流程将导致文档不准确、误导开发者，甚至引发生产事故。

### 流程概览图

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  Step 1:    │───▶│  Step 2:    │───▶│  Step 3:    │───▶│  Step 4:    │───▶│  Step 5:    │
│  读取文件    │    │  搜索代码    │    │  验证API     │    │  检查数据库  │    │  基于实施    │
│  Read Files  │    │ Search Code │    │ Verify API   │    │ Check DB    │    │ Implement   │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
       │                   │                   │                   │                   │
       ▼                   ▼                   ▼                   ▼                   ▼
  获取最新源码         定位实现位置         确认接口存在         验证字段真实         确保文档准确
```

### Step 1: 读取文件（Read Files）

**目标**: 获取最新的实际代码，而非依赖记忆或假设。

**操作清单**:
- [ ] 使用 `Read` 工具读取相关源文件（非缓存版本）
- [ ] 确认读取的是**当前分支**的最新代码
- [ ] 记录文件的**最后修改时间**和**作者**
- [ ] 如果涉及多个文件，**逐一读取**，不要遗漏

**验证要点**:
```markdown
## Step 1 检查记录
- 读取时间: YYYY-MM-DD HH:MM:SS
- 读取文件列表:
  - [x] src/views/QuizView.vue (最后修改: 2026-04-05)
  - [x] src/utils/shuffleOptions.js (最后修改: 2026-04-05)
  - [x] routes/questions.js (最后修改: 2026-04-04)
- 分支状态: main (commit: abc1234)
```

**典型错误**:
- ❌ 依赖记忆："我记得这个函数是这样写的"
- ❌ 读取旧版本：本地有未提交的修改但没读取到
- ❌ 遗漏关联文件：只读了前端没读后端

---

### Step 2: 搜索代码（Search Code）

**目标**: 通过搜索确认代码的实际实现，定位关键逻辑。

**操作清单**:
- [ ] 使用 `Grep` 工具搜索关键函数名、变量名、路由路径
- [ ] 使用 `SearchCodebase` 进行语义化搜索（理解代码意图）
- [ ] 使用 `Glob` 查找相关文件（确保没有遗漏）
- [ ] 搜索范围覆盖：前端组件、路由定义、服务层、数据库迁移脚本

**常用搜索模式**:
```bash
# 搜索函数定义
grep -rn "function shuffleOptions" src/

# 搜索路由定义
grep -rn "app\.\(get\|post\|put\|delete\)" routes/

# 搜索数据库字段
grep -rn "CREATE TABLE" migrations/
grep -rn "type.*ENUM" scripts/*.sql

# 搜索 Vue 组件中的关键逻辑
grep -rn "type === 'judgment'" src/views/ src/components/
```

**验证要点**:
- [ ] 搜索结果显示的代码与文档描述一致
- [ ] 没有找到**矛盾的实现**（同一功能多处不同实现）
- [ ] 确认搜索到了**所有相关位置**

**典型错误**:
- ❌ 只搜索了部分关键词
- ❌ 忽略了大小写差异
- ❌ 没有搜索注释掉的代码（可能是备选方案）

---

### Step 3: 验证 API（Verify API）

**目标**: 确认文档中描述的 API 端点确实存在且行为一致。

**操作清单**:
- [ ] 检查 `routes/` 目录下的路由定义文件
- [ ] 确认 HTTP 方法（GET/POST/PUT/DELETE）正确
- [ ] 确认 URL 路径完全匹配（包括前缀 `/api`）
- [ ] 验证请求参数名称和类型
- [ ] 验证响应数据结构（字段名、嵌套层级）
- [ ] 如有可能，使用 curl 或 Postman 实际调用测试

**API 验证模板**:
```markdown
## API 验证记录

### 接口: POST /api/questions/batch
- [x] 路由定义位置: routes/questions.js:Line 145
- [x] HTTP 方法: POST ✅
- [x] 认证要求: Bearer Token ✅
- [x] 请求体格式: JSON Array ✅
- [x] 响应结构: { code, msg, data: { total, success_count, fail_count, errors } } ✅
- [x] 实际测试: curl 测试通过 ✅
- [ ] 边界情况: 空数组 / 超过100条 / 无效题型
```

**典型错误**:
- ❌ 文档写了 `/api/question/add` 但实际是 `/api/questions`
- ❌ 文档说返回 `{ success: true }` 但实际返回 `{ code: 200, data: {...} }`
- ❌ 遗略了必需的请求头（如 Authorization, Content-Type）

---

### Step 4: 检查数据库（Check Database）

**目标**: 验证文档中描述的数据结构、字段名、约束条件与实际数据库一致。

**操作清单**:
- [ ] 读取 `migrations/` 目录下的 SQL 迁移脚本
- [ ] 检查 `config/database.js` 中的表结构定义（如有 ORM）
- [ ] 确认字段名、数据类型、约束（NOT NULL, DEFAULT, ENUM）
- [ ] 验证外键关系和索引定义
- [ ] 对比文档中的字段说明与实际 schema

**数据库验证模板**:
```markdown
## 数据库验证记录

### 表: questions
- [x] 迁移脚本: scripts/migrate-add-judgment-type.sql
- [x] type 字段: ENUM('single','multiple','judgment') NOT NULL DEFAULT 'single' ✅
- [x] options 字段: JSON 类型 ✅
- [x] answer 字段: VARCHAR(10) ✅
- [x] 外键: subject_id → subjects(id) ✅
- [x] 索引: idx_questions_type ✅

### 字段对比
| 文档描述 | 实际Schema | 匹配 |
|---------|-----------|------|
| type: ENUM(...) | ENUM('single','multiple','judgment') | ✅ |
| options: Array | JSON | ✅ |
| created_at: DateTime | TIMESTAMP | ✅ |
```

**典型错误**:
- ❌ 文档写 `type: VARCHAR(20)` 但实际是 `ENUM(...)`
- ❌ 文档遗漏了新增的字段（如 migration 后加了新 column）
- ❌ 枚举值拼写错误（`judgement` vs `judgment`）

### Step 5: 基于实施（Implement）

**目标**: 在完成以上 4 步验证后，基于真实的代码实现来编写或更新文档，而非凭空想象。

**操作清单**:
- [ ] 文档中的每个技术细节都有**代码依据**（标注文件路径和行号）
- [ ] 代码示例是从**实际代码复制**并适当简化，非手写
- [ ] 配置项、常量值与**源码一致**
- [ ] 版本号、状态标记反映**当前实际情况**
- [ ] 更新文档的**最后修改时间和版本号**

**实施原则**:
```markdown
## 文档编写规范

### 代码引用格式
根据实际代码（[shuffleOptions.js:Line 23-28](../../src/utils/shuffleOptions.js#L23-L28)）：

```javascript
// 实际代码
if (q.type === 'judgment') {
  return { ...q, options: q.options, shuffleMapping: null }
}
```

### 禁止事项
- ❌ "应该"、"大概"、"可能"等不确定词汇
- ❌ 未经验证的假设性描述
- ❌ 从其他项目复制的通用示例（除非明确标注）
- ❌ 过时的截图或命令输出
```

## 二、常见 AI 幻觉陷阱

### 2.1 陷阱分类

| 陷阱类型 | 描述 | 典型案例 | 风险等级 |
|----------|------|----------|----------|
| **先入为主** | 基于常见模式假设实现 | "注册就是账号+密码"（实际可能是手机号） | 🔴 高 |
| **过度推断** | 从有限信息推导不存在功能 | "肯定有分页功能"（实际没有） | 🔴 高 |
| **缺乏验证** | 未读代码就描述功能 | "这个 API 支持 XX 参数"（实际不支持） | 🔴 高 |
| **假设 API 存在** | 编造不存在的接口 | "DELETE /api/users/:id"（实际只有软删除） | 🟠 中高 |
| **混淆相似项目** | 混淆不同项目的实现 | 把 React 项目的写法写成 Vue 的 | 🟠 中 |
| **版本过时** | 描述旧版本的特性 | "使用 Options API"（实际已改 Composition API） | 🟡 中 |
| **语言/框架偏差** | 用熟悉的语法写示例 | 在 Vue 项目里写 JSX | 🟡 中 |
| **忽略边界情况** | 只描述正常流程 | 不提错误处理、权限检查 | 🟡 低 |

### 2.2 真实案例：判断题选项乱序 Bug

**背景**: 项目实现了判断题（judgment）题型，但出现了选项被随机打乱的 Bug。

**幻觉过程**（错误示范）:

```
❌ AI（未验证）:
"判断题的选项打乱功能应该在 QuizView.vue 的 shuffleOptions 函数中处理，
只需要加一个 if 判断 type === 'judgment' 就行了。"

→ 结果：去 QuizView.vue 找不到 shuffleOptions，
    实际是在 utils/shuffleOptions.js 工具函数中。
    并且还遗漏了后端的验证逻辑。
```

**正确过程**（遵循 5 步法）:

```
✅ Step 1: 读取文件
   - Read: src/utils/shuffleOptions.js → 发现确实缺少判断题检查
   - Read: src/views/QuizView.vue → 看到调用了 shuffleOptions

✅ Step 2: 搜索代码
   - Grep: "shuffleOptions" → 找到 3 处调用
   - Grep: "type === 'judgment'" → 只有 1 处（在 QuizView 渲染时）

✅ Step 3: 验证 API
   - Read: routes/questions.js → 发现 validateJudgmentQuestion 中间件
   - 确认后端已有 options 校验

✅ Step 4: 检查数据库
   - Read: scripts/migrate-add-judgment-type.sql
   - 确认 type 字段 ENUM 包含 'judgment'

✅ Step 5: 基于实施
   - 文档明确指出修复位置：shuffleOptions.js:Line 15
   - 提供完整的修复代码（从实际代码复制修改）
   - 补充后端验证逻辑说明
```

**修复方案**（有代码依据）:
```javascript
// src/utils/shuffleOptions.js (基于实际代码 Line 12-25)
export function shuffleOptions(question) {
  // 🔒 关键修复：判断题跳过打乱
  // 验证来源: 已读取该文件，确认此处缺少判断
  if (question.type === 'judgment') {
    return {
      ...question,
      options: question.options || ['对', '错'],
      shuffleMapping: null
    }
  }

  // 原有的打乱逻辑...
  const shuffled = [...question.options].sort(() => Math.random() - 0.5)
  const mapping = generateMapping(question.options, shuffled)

  return {
    ...question,
    options: shuffled,
    shuffleMapping: mapping
  }
}
```

## 四、质量标准

### 4.1 文档质量三要素

| 要素 | 定义 | 验证方法 | 达标标准 |
|------|------|----------|----------|
| **准确性** | 技术细节与代码完全一致 | 5步法验证 | 零错误 |

### 4.2 代码依据要求

文档中的每个技术断言都必须满足以下之一：

**A级依据（强推荐）**:
- 直接引用源代码（附文件路径和行号）
- 附带完整的代码片段（从源码复制）
- 包含实际的 API 响应示例（来自测试环境）

**B级依据（可接受）**:
- 基于多个文件的综合分析结论
- 经过验证的架构图/流程图
- 来自官方文档且已验证版本兼容性

**C级依据（需标注）**:
- 基于行业最佳实践的通用建议
- 待验证的设计方案（标注"草案"或"计划"）
- 第三方工具/库的使用说明（附版本号）

**禁止（不合格）**:
- ❌ "应该是这样"的猜测
- ❌ 从其他项目复制的未验证内容
- ❌ 与代码矛盾的描述

## 六、更新后的文档验证命令

完成文档更新后，执行以下命令进行最终验证：

### 6.1 自动化验证脚本

```bash
#!/bin/bash
# verify-docs.sh - 文档验证脚本
# 用途: 检查文档中的链接有效性、代码引用准确性

echo "=========================================="
echo "  文档自动化验证工具"
echo "  版本: 1.0 | 日期: 2026-04-05"
echo "=========================================="

# 1. 检查 Markdown 链接
echo ""
echo "[1/5] 检查文档内部链接..."
find DOCS -name "*.md" -exec grep -Ho '\[.*\](.*)' {} \; | \
  grep -oP '(?<=\()[^)]+(?=\))' | \
  while read link; do
    if [[ $link == http* ]]; then
      # 外部链接：仅报告，不验证（需要网络）
      echo "  外部链接: $link"
    elif [[ ! -f "$link" ]] && [[ ! -d "$link" ]]; then
      echo "  ❌ 断链: $link"
    fi
  done

# 2. 检查代码文件引用
echo ""
echo "[2/5] 验证代码文件引用..."
grep -rh '](.*\.js)' DOCS/ --include="*.md" | \
  grep -oP '(?<=\().*?\.js(?=\))' | sort -u | \
  while read file; do
    if [[ ! -f "$file" ]]; then
      echo "  ❌ 文件不存在: $file"
    else
      echo "  ✅ $file"
    fi
  done

# 3. 检查路由引用
echo ""
echo "[3/5] 验证 API 路径引用..."
grep -rh '`POST\|`GET\|`PUT\|`DELETE' DOCS/ --include="*.md" | \
  grep -oP '`[A-Z]+ /api[^`]+' | sort -u | \
  while read route; do
    method=$(echo $route | grep -oP '[A-Z]+')
    path=$(echo $route | grep -oP '/api[^`]+')
    if grep -rq "$method.*$path" routes/; then
      echo "  ✅ $route"
    else
      echo "  ⚠️  可能不存在: $route"
    fi
  done

# 4. 检查 SQL 表名引用
echo ""
echo "[4/5] 验证数据库表名引用..."
grep -rh '表 \`?\w+\`?' DOCS/ --include="*.md" | \
  grep -oP '(?:表 )?[`\']?(\w+)[`\']?' | \
  while read table; do
    if grep -rq "CREATE TABLE.*$table\|$table" scripts/ migrations/; then
      echo "  ✅ 表 $table 存在"
    else
      echo "  ⚠️  表 $table 未找到定义"
    fi
  done

# 5. 生成验证报告
echo ""
echo "[5/5] 生成验证报告..."
REPORT_FILE="docs-verification-$(date +%Y%m%d-%H%M%S).log"
echo "验证完成时间: $(date)" > $REPORT_FILE
echo "文档总数: $(find DOCS -name '*.md' | wc -l)" >> $REPORT_FILE

echo ""
echo "=========================================="
echo "  验证完成！详细报告: $REPORT_FILE"
echo "=========================================="
```

## 七、应急处理流程

### 7.1 发现文档错误后的处理

```
发现文档错误
    ↓
立即评估影响范围
    ├── 低风险（笔误、格式问题）
    │   └── 直接修正，记录到 Changelog
    │
    ├── 中风险（误导但不导致故障）
    │   └── 1小时内修正，发送更正通知
    │
    └── 高风险（可能导致生产事故）
        ├── 立即标记文档为 "⚠️ 待验证"
        ├── 通知所有相关人员停止参考该文档
        ├── 启动 5 步法紧急验证
        └── 修正后走完整审核流程
```

```

## 八、总结与承诺

### 核心原则

> **"没有代码依据的文档，就是技术债务。"**

每一行文档都应该能够追溯到具体的代码实现。当我们编写文档时，我们不是在创作文学作品，而是在创建**可执行的、可验证的技术规范**。

### 个人承诺

作为本文档的维护者和使用者，我承诺：

1. **永远不凭记忆写文档** - 每次都读取最新代码
2. **永远不假设 API 存在** - 每次都验证路由定义
3. **永远不省略验证步骤** - 即使是"小修改"也完整执行
4. **永远不发布未验证的内容** - 宁可慢一点，也要准一点
5. **发现错误立即修正** - 不掩盖、不拖延、不推诿
