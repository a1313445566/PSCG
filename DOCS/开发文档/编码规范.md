# 编码规范

## 1. 前端编码规范

### 1.1 Vue 组件规范

#### 组件命名
- **组件名**: 使用 PascalCase 命名（如 `QuestionCard.vue`）
- **文件名**: 与组件名保持一致
- **目录结构**: 按功能模块组织组件

#### 模板规范
- 使用 `<script setup>` 语法
- 模板中的变量和方法使用 camelCase
- 模板表达式简洁明了，复杂逻辑应提取到 script 部分

#### 脚本规范
- **变量命名**: 使用 camelCase
- **常量命名**: 使用 UPPER_SNAKE_CASE
- **方法命名**: 使用 camelCase，动词开头
- **Props**: 必须定义类型和默认值
- **响应式数据**: 使用 `ref()` 或 `reactive()`
- **计算属性**: 使用 `computed()`
- **生命周期**: 使用组合式 API 的生命周期钩子

#### 样式规范
- 使用 scoped 样式
- 缩进使用 2 空格
- 支持 SCSS
- 选择器使用 kebab-case
- 避免使用 ID 选择器

### 1.2 JavaScript 规范

#### 变量和函数
- **变量声明**: 使用 `const` 或 `let`，避免使用 `var`
- **函数定义**: 使用箭头函数或函数表达式
- **参数命名**: 使用 camelCase
- **返回值**: 函数应有明确的返回值类型

#### 代码风格
- 缩进使用 2 空格
- 行尾不使用分号
- 字符串使用单引号
- 对象字面量使用简洁语法
- 数组和对象的括号与内容之间有空格

#### 错误处理
- 所有异步操作必须使用 try/catch
- 使用统一的错误提示机制
- 错误信息应清晰明了

## 2. 后端编码规范

### 2.1 Node.js 规范

#### 文件结构
- **路由文件**: 放在 `routes/` 目录
- **服务文件**: 放在 `services/` 目录
- **中间件**: 放在 `middleware/` 目录
- **工具函数**: 放在 `utils/` 目录

#### 命名规范
- **文件命名**: 使用 kebab-case（如 `user-stats.js`）
- **变量命名**: 使用 camelCase
- **常量命名**: 使用 UPPER_SNAKE_CASE
- **函数命名**: 使用 camelCase，动词开头
- **类命名**: 使用 PascalCase

#### 代码风格
- 缩进使用 2 空格
- 行尾不使用分号
- 字符串使用单引号
- 对象字面量使用简洁语法
- 数组和对象的括号与内容之间有空格

#### 错误处理
- 所有异步操作必须使用 try/catch
- 统一的错误响应格式
- 错误信息应清晰明了

### 2.2 API 规范

#### 路由规范
- 使用 RESTful 风格
- 路由路径使用 kebab-case
- 动词使用 HTTP 方法（GET, POST, PUT, DELETE）

#### 响应格式
- 统一的 JSON 响应格式：
  - 成功响应：直接返回数据或 `{ success: true, data: {} }`
  - 错误响应：`{ error: '错误信息' }` 或 HTTP 状态码 + 错误信息
- HTTP 状态码：
  - 200: 成功
  - 400: 请求参数错误
  - 401: 未授权
  - 403: 禁止访问
  - 404: 资源不存在
  - 500: 服务器内部错误

#### 参数验证
- 所有请求参数必须验证
- 使用参数化查询防止 SQL 注入
- 验证失败返回明确的错误信息
- 参考 `services/validationService.js` 中的验证逻辑

## 3. 数据库规范

### 3.1 SQL 规范
- 使用 MySQL 语法
- 表名使用 snake_case
- 字段名使用 snake_case
- 主键使用 `id`，自增
- 外键使用 `{table_name}_id` 格式
- 禁止使用 `SELECT *`，必须指定字段
- 使用参数化查询防止 SQL 注入

### 3.2 索引规范
- 高频查询字段添加索引
- 外键字段添加索引
- 复合索引应考虑查询顺序

### 3.3 事务规范
- 涉及多表操作时使用事务
- 事务应包含尽可能少的操作
- 事务应及时提交或回滚

## 4. 通用规范

### 4.1 注释规范
- 关键逻辑必须添加中文注释
- 函数应添加 JSDoc 注释
- 复杂算法应添加详细注释
- 避免冗余注释

### 4.2 版本控制
- 使用 Git 进行版本控制
- 提交信息应清晰明了
- 提交信息格式：`[类型] 描述`
  - 类型：feat, fix, docs, style, refactor, test, chore
- 避免提交敏感信息

### 4.3 性能规范
- **前端**: 组件懒加载，路由懒加载，图片优化
- **后端**: 数据库索引，连接池复用，缓存机制
- **API**: 防抖，超时处理，自动重试
- **大数据**: 批量操作分批执行，每次 ≤ 1000 条

### 4.4 安全规范
- **SQL 注入防护**: 使用参数化查询
- **XSS 防护**: 输入验证，输出转义
- **CSRF 防护**: 使用 CSRF token
- **认证授权**: JWT 认证，基于角色的权限控制
- **敏感信息**: 使用环境变量，不在代码中硬编码

## 7. 代码审查标准

### 7.1 审查要点
- **功能正确性**: 代码是否实现了预期功能
- **代码质量**: 是否符合编码规范
- **性能优化**: 是否存在性能问题
- **安全问题**: 是否存在安全隐患

- 提交代码前自我审查
- 团队成员代码审查
- 解决审查中发现的问题
- 确认通过后合并代码

## 8. 常见问题

### 8.1 代码风格问题

**问题**: 缩进不一致  
**解决方案**: 使用 Prettier 自动格式化
```bash
npm run format
```

**问题**: 命名不规范  
**解决方案**: 遵循命名规范，使用有意义的变量名
- 组件/页面：PascalCase
- 变量/方法：camelCase
- 常量：UPPER_SNAKE_CASE

**问题**: 代码冗余  
**解决方案**: 提取重复代码，使用函数或组件复用

### 8.2 性能问题

**问题**: 频繁渲染  
**解决方案**: 使用 `computed` 和 `watch` 优化
- 避免在模板中进行复杂计算
- 使用计算属性缓存结果

**问题**: 数据库查询缓慢  
**解决方案**: 添加索引，优化查询语句
- 高频查询字段添加索引
- 避免 `SELECT *`，只查询需要的字段
- 使用 `EXPLAIN` 分析查询

**问题**: API 响应缓慢  
**解决方案**: 使用缓存，优化业务逻辑
- 使用 `services/cache.js` 缓存热点数据
- 批量操作分批执行

### 8.3 安全问题

**问题**: SQL 注入  
**解决方案**: 使用参数化查询
- 所有数据库查询使用参数化
- 参考 `services/database.js` 中的参数化查询示例

**问题**: XSS 攻击  
**解决方案**: 输入验证，输出转义
- 使用 `utils/xss-filter.js` 过滤用户输入
- 富文本内容使用白名单过滤

**问题**: 敏感信息泄露  
**解决方案**: 使用环境变量，加密存储
- 数据库密码等敏感信息使用环境变量
- API 签名密钥使用 `SIGNATURE_SECRET`
- 不在代码中硬编码敏感信息
