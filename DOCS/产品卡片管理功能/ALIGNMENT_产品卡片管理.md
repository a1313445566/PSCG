## 四、需求理解

### 4.1 现状分析

#### 当前产品卡片实现位置
**文件**: `src/views/NewHomeView.vue:53-96`

```javascript
const productCards = [
  {
    title: '课程管理',
    description: '让课程管理更简单高效',
    icon: Reading,           // Element Plus 图标组件
    iconClass: '',          // CSS 类名
    tag: null               // 标签（如 'hot'）
  },
  // ... 共 6 个卡片
]
```

#### 关键问题
1. ❌ **数据硬编码**：产品卡片数据写死在前端代码中
2. ❌ **无法动态修改**：每次修改都需要重新部署前端代码
3. ❌ **缺少跳转链接**：当前没有 route/link 字段
4. ❌ **无后台管理**：管理员无法通过界面管理这些卡片

### 4.2 目标状态

#### 改造后的产品卡片数据流
```
数据库 (product_cards 表)
    ↓
后端 API (/api/product-cards)
    ↓
前端 NewHomeView.vue（从 API 动态加载）
    ↓
渲染产品卡片（支持点击跳转）
```

#### 后台管理流程
```
AdminView.vue → ProductCardManagement.vue（新增组件）
    ↓
调用后端 CRUD API
    ↓
增删改查产品卡片配置
```

### 4.3 技术方案预判

#### 数据库设计思路
```sql
CREATE TABLE product_cards (
  id INT PRIMARY KEY AUTO_INCREMENT,
  title VARCHAR(100) NOT NULL,          -- 标题
  description VARCHAR(200),             -- 副标题
  icon_name VARCHAR(50) NOT NULL,       -- 图标名称（如 'Reading'）
  icon_class VARCHAR(50),               -- 图标样式类
  link_type ENUM('route', 'url'),       -- 链接类型
  link_value VARCHAR(255),              -- 跳转值（路由路径或URL）
  tag VARCHAR(20),                      -- 标签（hot/new 等）
  sort_order INT DEFAULT 0,             -- 排序序号
  is_visible TINYINT(1) DEFAULT 1,      -- 是否可见
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

#### API 设计思路
```
GET    /api/product-cards              # 获取所有可见卡片（前台使用）
GET    /api/admin/product-cards        # 获取所有卡片（后台管理）
POST   /api/admin/product-cards        # 创建卡片
PUT    /api/admin/product-cards/:id    # 更新卡片
DELETE /api/admin/product-cards/:id    # 删除卡片
PUT    /api/admin/product-cards/sort   # 批量更新排序
```

#### 前端组件设计思路
参考现有的 `NavigationManagement.vue` 模式：
- 使用 `el-table` 展示卡片列表
- 使用 `el-dialog` 作为编辑对话框
- 使用 `el-form` + `el-input` 等组件进行表单编辑
- 图标选择器：下拉选择或图标面板

### 5.1 需要确认的关键决策点

#### 🔴 决策点 1：图标存储方式
**问题**：如何存储和显示 Element Plus 图标？

| 方案 | 优点 | 缺点 |
|------|------|------|
| **A. 存储图标名称字符串** | 实现简单，数据库友好 | 需要在前端维护图标名称到组件的映射表 |
| **B. 存储图标 SVG 字符串** | 灵活度高 | 数据库存储冗余，维护成本高 |
| **C. 使用图标索引** | 性能最优 | 不直观，扩展性差 |

**推荐方案**：A（存储图标名称字符串）
- 理由：Element Plus 图标有固定的命名规范（如 `Reading`, `UserFilled`）
- 实现：在前端维护一个图标映射对象或动态导入机制

**待确认**：是否采用方案 A？

**推荐方案**：C（同时支持内部路由和外部 URL）

## 七、风险提示

### 7.1 技术风险
- ⚠️ **图标动态导入性能**：如果图标数量较多，可能影响首屏加载
  - **缓解措施**：按需加载或使用虚拟滚动
- ⚠️ **前后端数据一致性**：前台缓存可能导致更新延迟
  - **缓解措施**：设置合理的缓存策略或实时刷新

### 7.2 业务风险
- ⚠️ **误操作风险**：管理员可能误删重要卡片
  - **缓解措施**：删除前二次确认 + 提供恢复机制
- ⚠️ **链接失效风险**：配置的跳转链接可能失效
  - **缓解措施**：定期检查链接有效性（可选）
