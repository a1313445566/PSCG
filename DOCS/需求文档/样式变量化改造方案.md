## 一、改造概述

### 1.1 目标

将项目约 **1500 处**硬编码替换为 SCSS/CSS 变量，消除技术债，为主题切换奠基。

### 1.2 根因

| 原因 | 影响 |
|------|------|
| Vite 未配置 `additionalData` | `.vue` 文件无法直接用 `$primary-color` 等 SCSS 变量，被迫硬编码 |
| 缺渐变/按钮阴影等组合变量 | 同一值反复手写 |
| `_game.scss`(996行) + `_admin.scss`(577行) 自身含 170+ 处硬编码 | 榜样效应差 |
| 无防回归机制 | 技术债持续累积 |

### 1.3 原则

1. **先修根因再动手术** — 先配 `additionalData`，阻止新硬编码
2. **复用现有架构** — 在 `_variables.scss` 追加，不新建文件
3. **完全兼容** — 零视觉变化
4. **两域分治** — 用户端与后台域独立推进，互不干扰
5. **渐进式** — 各批次独立可回滚
6. **可维护** — 建多层防回归机制
7. **瘦身先行** — 样式超标文件在变量化改造时同步抽离为独立 `.scss`，不单独开专项

## 三点五、样式拆分专项（与变量化同步推进）

> 这是一个独立但高度关联的治理目标：**将 `.vue` 文件中臃肿的 `<style>` 块抽离为同名 `.scss` 文件**，让每个 `.vue` 文件只专注逻辑与模板。

### 3.5.1 现状诊断

实测各文件样式占比（2026-04-06）：

| 文件 | 总行数 | 样式行数 | 样式占比 | 优先级 |
|------|--------|---------|---------|--------|
| `LeaderboardTable.vue` | 528 | 414 | **78%** | 🔴 P0 |
| `HomeView.vue` | 1259 | 867 | **69%** | 🔴 P0 |
| `DocsView.vue` | 1436 | 901 | **63%** | 🔴 P0 |
| `LeaderboardView.vue` | 2096 | 1167 | **56%** | 🔴 P0 |
| `LearningProgressView.vue` | 720 | 442 | **61%** | 🔴 P0 |
| `LoginForm.vue` | 1316 | 591 | **45%** | 🟠 P1 |
| `BackupRestore.vue` | 897 | 380 | **42%** | 🟠 P1 |
| `DataCleanup.vue` | 666 | 283 | **42%** | 🟠 P1 |
| `QuizView.vue` | 1142 | 248 | 22% | 🟡 P2 |
| `AnswerHistoryView.vue` | 678 | 201 | 30% | 🟡 P2 |

> **结论**：重灾区的样式行数动辄是逻辑代码的 2 倍，极大降低了文件可读性和维护效率。

### 3.5.2 拆分规则

**触发条件**（满足任意一条即应拆分）：
- 样式超过 **200 行** 且占比 > **30%**
- 文件总行数超过 **600 行**

**命名约定**：

```
src/
  views/
    LeaderboardView.vue          ← 只保留 <style src="./LeaderboardView.scss" scoped />
    LeaderboardView.scss         ← 抽离的样式文件（与 .vue 同目录）
  components/
    leaderboard/
      LeaderboardTable.vue
      LeaderboardTable.scss
```

**`.vue` 文件中保留的内容**（不抽离）：
- 少于 20 行的极简样式
- 组件级 CSS 变量（`:host` / `:root` 局部覆盖）
- 必须内联的 `:deep()` 动态绑定

**抽离文件的写法**：

```vue
<!-- 抽离前 -->
<style lang="scss" scoped>
  /* 900 行样式 */
</style>

<!-- 抽离后 -->
<style lang="scss" src="./HomeView.scss" scoped></style>
```

```scss
// HomeView.scss — 顶部声明，无需 @use（因为 vite additionalData 已注入）
// 若使用工具函数，则需显式：
// @use "@/styles/scss/abstracts/_functions" as *;

.home-view {
  // ...
}
```

### 3.5.3 与变量化改造的协同策略

**关键原则：不单独开专项，变量化时同步抽离。**

改造某个文件时，按以下顺序一次性完成：

```
Step 1  变量化替换（硬编码 → $variable）
Step 2  抽离 <style> 块为同名 .scss 文件
Step 3  .vue 改为 <style lang="scss" src="./Xxx.scss" scoped>
Step 4  视觉回归验证
Step 5  单独 Git commit（"refactor(style): LeaderboardView 变量化+样式抽离"）
```

> 这样一次改动同时消灭两个问题，避免重复打开同一文件两次。

### 3.5.4 目标目录结构（改造完成后）

```
src/
├── views/
│   ├── LeaderboardView.vue       # ~930 行 → ~400 行（只剩 template + script）
│   ├── LeaderboardView.scss      # ~1167 行，纯样式，变量化后约 700 行
│   ├── HomeView.vue              # ~1259 行 → ~400 行
│   ├── HomeView.scss             # ~867 行，变量化后约 550 行
│   ├── DocsView.vue              # ~1436 行 → ~550 行
│   ├── DocsView.scss             # 独立样式文件
│   └── ...（其余超标文件同理）
├── components/
│   ├── leaderboard/
│   │   ├── LeaderboardTable.vue  # ~528 行 → ~120 行
│   │   └── LeaderboardTable.scss # 纯样式
│   ├── common/
│   │   ├── LoginForm.vue
│   │   └── LoginForm.scss
│   └── admin/
│       ├── BackupRestore.vue
```

### 3.5.5 防反弹机制

在 `scripts/style-size-monitor.js` 中扩展现有监控脚本，增加：

| 检测项 | 阈值 | 告警级别 |
|--------|------|---------|
| `.vue` 文件样式行数 | > 200 行 | ⚠️ WARNING |
| `.vue` 文件样式行数 | > 400 行 | 🔴 ERROR（阻断 CI）|
| `.vue` 文件总行数 | > 800 行 | ⚠️ WARNING |

命令集成：

```json
// package.json
"scripts": {
  "monitor:style-size": "node scripts/style-size-monitor.js",
  "monitor:style-size:check": "node scripts/style-size-monitor.js --ci"
}
```

## 八、Phase 5：扫尾 + 自动化基础设施（3-4 h）

### 8.1 视觉回归测试

使用 Playwright `toHaveScreenshot` 进行像素级 diff：

- **基线截图**：Phase 0 前对 10 个用户页 + 3 个后台页 × 3 视口（desktop/tablet/mobile）拍摄基线
- **容差策略**：全局默认 1% 像素差异；图表区域 5%；文字区域 0.1%
- **工作流**：每 Phase 结束后运行 `npm run test:visual`，自动检测视觉偏移
- **CI 集成**：GitHub Actions push 时自动触发（可选）

> 完整测试文件：`tests/visual/snapshot.spec.js`

### 8.2 硬编码监控脚本

| 命令 | 用途 |
|------|------|
| `npm run monitor:scan` | 全量扫描 + TOP15 排行榜 |
| `npm run monitor:baseline` | 记录当前快照为基线 |
| `npm run monitor:diff` | 与基线对比，显示恶化/改善趋势 |
| `npm run monitor:check` | CI 模式（退出码 0/1）|

支持 pre-commit hook 和 GitHub Actions 集成。
扫描范围：`src/views/` + `src/components/` 中 `.vue` 文件的 `<style>` 块，检测 hex/rgba/gradient/shadow/named-color 六类模式，加权评分，分级告警。

> 完整脚本：`scripts/hardcoded-monitor.js`

### 8.3 可接受例外

| 类型 | 示例 |
|------|------|
| SVG 内联 fill/stroke | `KawaiiCharacter.vue` |
| Element Plus 深度覆盖 | 必须用库规定格式 |
| 功能性语义色 | `status-tag success/warning` 等 |
| 透明度黑/白 rgba | `rgba(0,0,0,0.x)` 过于通用 |
| template 中动态 `:style` | JS 表达式计算的颜色 |

### 8.4 防回归体系（6 层）

| 层级 | 工具 | 时机 |
|------|------|------|
| L1 | Stylelint | 编码时 |
| L2 | Pre-commit Hook (`monitor:check`) | 每次 commit |
| L3 | Playwright 截图对比 | Push / Phase 结束 |
| L4 | CI Pipeline | PR |
| L5 | Cron 定期巡检 | 每日/周 |
| L6 | Code Review Checklist | CR 时 |

**至少部署 L1+L2+L3 三层。**

## 十二、风险与应对

| 风险 | 概率 | 应对 |
|------|------|------|
| 实际工作量超统计 | 高 | 每批次预留 20% 缓冲；低优先级留下一轮 |
| `set-alpha()/tint()/shade()` 编译报错 | 中 | Vue style 中显式 `@use _functions`; 或改用预计算的 CSS 变量 |
| `additionalData` 拖慢编译 | 低 | 注入纯变量文件 <150 行无副作用；Vite HMR 增量编译 |
| 新变量意外覆盖局部样式 | 低 | 语义命名 + 每文件立即视觉验证 + 分批提交可回滚 |
| Stage A/B 并行引发变量命名冲突 | 低 | `$admin-*` 前缀隔离；两路改 `_variables.scss` 需合并时注意顺序 |
| 浏览器兼容 | 极低 | 目标浏览器均原生支持 CSS 变量；SCSS 编译时替换零风险 |
