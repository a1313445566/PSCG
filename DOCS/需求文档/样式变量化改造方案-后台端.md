## 一、后台端改造概述

### 1.1 目标范围

| 域 | 文件范围 | 硬编码估算 |
|----|---------|-----------|
| 后台主题 | `_admin.scss`（577 行） | ~50+ 处 |
| 后台 Views | `AdminView.vue` + `admin/DashboardView.vue` | ~52 处 |
| 后台 Components | `src/components/admin/` 下 41 个 Vue 组件 | ~230 处 |
| **合计** | | **~330 处** |

> 后台端工作量约为用户端的 **1/3**，但组件数量多（41 个），建议按功能模块分批处理。

### 1.2 根因（两端共有）

| 原因 | 影响 |
|------|------|
| Vite 未配置 `additionalData` | `.vue` 文件无法直接使用 `$admin-bg` 等 SCSS 变量，被迫硬编码 |
| 缺少 `$admin-*` 专用变量体系 | Admin 色系（`#303133` / `#f5f7fa` / `#ebeef5` 等）全部硬写 |
| `_admin.scss` 自身含 50+ 处硬编码 | 主题文件带头破坏规范 |
| 无防回归机制 | 41 个 admin 组件改完后容易被新代码污染 |

### 1.3 原则

1. **前置已就绪再开工**：Phase 0（Vite 修复）和 Phase 1（变量补齐）必须由用户端或本端先行完成
2. **`$admin-*` 前缀隔离**：所有后台专用变量统一使用 `$admin-` 前缀，避免与用户端变量冲突
3. **Element Plus 深度覆盖不动**：`:deep(.el-*)` 内部保持库规定格式，不强行替换
4. **零视觉变化**：每文件改完即视觉快照验证
5. **按功能模块分批提交**：41 个组件分 3 批，每批可独立回滚

#### 如需补充更多变量

在实施过程中，若遇到未定义的变量：

1. **先检查 `_variables.scss`** —— 确认是否已存在
2. **若不存在，按命名规范追加**：
   ```scss
   // 在对应分类下追加
   $new-variable-name: #color-value;
   ```
3. **同步 CSS 变量**（用于内联样式）：
   ```scss
   :root {
     --new-variable-name: #{$new-variable-name};
   }
   ```

**命名规范**：
- 后台专用：`$admin-{property}`
- 背景：`$bg-{level}-{hue}`
- 边框：`$border-{context}`
- 阴影：`$shadow-{context}`

## 四、样式文件抽离规则（与变量化同步推进）

### 4.1 触发条件

满足**任意一条**即必须在本次改造时抽离：

| 条件 | 阈值 |
|------|------|
| `<style>` 行数 | > 200 行 且占比 > 30% |
| 文件总行数 | > 600 行 |

### 4.2 后台组件抽离方式

```
变量化替换 → 抽离 <style> 为同名 .scss → .vue 改用 src 引用 → 视觉验证 → 单次 commit
```

```vue
<!-- 改造后的 admin 组件 -->
<style lang="scss" src="./styles/BackupRestore.scss" scoped></style>
```

```
src/components/admin/
  styles/                     ← admin 组件样式目录
    BackupRestore.scss
    DataCleanup.scss
    ... （其余超标组件）
  BackupRestore.vue            ← 只剩 template + script
  DataCleanup.vue
  ...
```

**特别注意**：admin 组件大量使用 Element Plus，抽离时注意 `:deep(.el-*)` 选择器在独立文件中仍然有效（scoped 作用域由 `src` attribute 继承）。

### 4.3 防反弹

```bash
npm run monitor:style-size        # 检查所有 .vue 样式行数
npm run monitor:style-size:check  # CI 模式，样式 > 400 行时退出码 1
```

| `#ff6b6b` | `$primary-color` |
| `#06d6a0` | `$success-color` |
| `#ef476f` | `$danger-color` |
| `#118ab2` | `$info-color` |
| `#ffd166` | `$accent-color` |

**阴影**

| 硬编码值 | 替换为 |
|---------|--------|
| `0 2px 12px rgba(0,0,0,0.1)` | `$admin-data-card-shadow` |
| `0 4px 20px rgba(0,0,0,0.15)` | `$admin-data-card-shadow-hover` |
| `0 2px 8px rgba(0,0,0,0.08)` | `$admin-table-shadow` |
| `0 2px 8px rgba(0,0,0,0.1)` | `$shadow-sm`（通用小阴影）|

**RGBA 函数替换**

| 硬编码模式 | 替换为 |
|-----------|--------|
| `rgba(255,107,107, 0.x)` | `set-alpha($primary-color, x*100)` |
| `rgba(0,0,0, 0.x)` 作为阴影色 | 优先用具名阴影变量；过于通用可保留 |

> **注意**：在独立 `.scss` 文件中使用 `set-alpha()` 等函数时需显式声明：
> ```scss
> @use "@/styles/scss/abstracts/_functions" as *;
> ```

## 八、Phase 5：扫尾 + 自动化基础设施（3-4 h，两端共用）

> Phase 5 在 Stage A 和 Stage B **均完成后**执行一次，统一收尾。

### 8.1 视觉回归测试（统一基线）

使用 Playwright `toHaveScreenshot` 进行像素级 diff：

| 快照范围 | 页面 | 视口 |
|---------|------|------|
| 用户端 | 首页、排行榜、测验页、学习进度、文档页、档案页 | desktop / tablet / mobile |
| 后台端 | 登录后 Dashboard、用户管理、题目管理、备份/清理页 | desktop / tablet |

- **容差策略**：全局 1%；图表区域 5%；纯文字区域 0.1%
- **运行时机**：每 Phase 结束后 `npm run test:visual`
- **CI 集成**：push 时自动触发（可选）

> 完整测试文件：`tests/visual/snapshot.spec.js`

### 8.2 硬编码监控脚本

```bash
npm run monitor:scan       # 全量扫描 + TOP15 排行榜
npm run monitor:baseline   # 记录当前快照为基线
npm run monitor:diff       # 与基线对比，显示恶化/改善趋势
npm run monitor:check      # CI 模式（退出码 0/1）
```

### 8.3 样式体积监控

```bash
npm run monitor:style-size        # 输出所有 .vue 样式行数报告
npm run monitor:style-size:check  # CI 模式，样式 > 400 行时退出码 1
```

### 8.4 防回归体系（6 层）

| 层级 | 工具 | 时机 |
|------|------|------|
| L1 | Stylelint（禁用硬编码色规则） | 编码时实时提示 |
| L2 | Pre-commit Hook（`monitor:check`） | 每次 `git commit` |
| L3 | Playwright 截图对比 | 每次 push / Phase 结束 |
| L4 | CI Pipeline（`monitor:check` + `style-size:check`） | PR 合并前 |
| L5 | Cron 定期巡检（`monitor:diff`） | 每日 / 每周 |
| L6 | Code Review Checklist | CR 时人工确认 |

**至少部署 L1 + L2 + L3 三层。**

### 8.5 可接受例外清单

| 类型 | 示例 | 说明 |
|------|------|------|
| SVG 内联 `fill`/`stroke` | `KawaiiCharacter.vue` | 图形颜色数据，不属于 CSS 样式 |
| Element Plus 深度覆盖 | `:deep(.el-button)` 内部 | 必须遵循库规定格式 |
| 功能性语义色 | `status-tag.success/warning` | 语义本身就是颜色含义 |
| 透明度纯黑/白 `rgba` | `rgba(0,0,0,0.x)` | 过于通用，替换收益低 |
| template 动态 `:style` | `:style="{ color: score > 90 ? '#06d6a0' : '#ef476f' }"` | JS 表达式颜色不在本次范围 |

## 十、风险与应对

| 风险 | 概率 | 应对 |
|------|------|------|
| 41 个 admin 组件实际量超统计 | 高 | 批次 3 预留 30% 缓冲；低密度组件快速扫描跳过 |
| Element Plus `:deep()` 覆盖被意外修改 | 中 | 改造前建立快照基线；`:deep()` 内部原则上不动 |
| `$admin-*` 变量与 Element Plus 默认值冲突 | 低 | `$admin-text-primary: #303133` 与 Element Plus 默认值一致，无冲突 |
| 独立 `.scss` 中 `set-alpha()` 编译失败 | 中 | 显式 `@use "_functions" as *`；或直接写 `rgba()` 预计算值 |
| Stage A/B 并行时同时修改 `_variables.scss` | 低 | `$admin-*` 前缀隔离，合并时按 admin 块追加，不覆盖 |
| 批次提交顺序错误导致样式闪烁 | 极低 | 按 Phase 2B → 3B → 4B 顺序，不跳批次 |
